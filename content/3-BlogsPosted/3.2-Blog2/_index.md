# DYNAMIC IMAGE PROCESSING ON AWS WITH AMAZON SAGEMAKER AND AMAZON BEDROCK

Leveraging Generative AI to build creative assisting tools requires addressing two key challenges: selecting the appropriate machine learning models and optimizing the inference infrastructure to minimize operational costs. 

A two-part AWS blog series details a hybrid solution for building an intelligent image editing tool (Inpaint Eraser) that combines Text-Guided Inpainting and Outpainting using Amazon SageMaker and Amazon Bedrock.

---

### Part 1: Building an AI Inpaint Eraser on Amazon SageMaker MME

The goal of this part is to create an intelligent object removal tool (Inpaint Eraser) that operates in two steps: generating a segmentation mask based on user click coordinates, and performing inpainting to fill in the background details naturally.

#### Machine Learning Models Used:
* **Segment Anything Model (SAM) by Meta:** A foundation model trained on the SA-1B dataset (11M images, 1.1B masks). SAM provides outstanding zero-shot generalization capabilities, allowing it to isolate and create a segmentation mask for any object based on click coordinate inputs without needing retraining.
* **Resolution-robust Large Mask Inpainting with Fourier Convolutions (LaMa):** Unlike traditional inpainting models that involve complex multi-step pipelines, LaMa uses a single-step architecture based on Fast Fourier Convolutions (FFCs). This design allows the model to capture global context across the entire image instantly, delivering natural results and faster inference speeds.

#### AWS Infrastructure Architecture:
* **AWS SageMaker Multi-Model Endpoints (MME) on GPU (Amazon EC2 G5 Instances):** To optimize hardware costs, the system avoids running separate GPU instances for each model. Instead, SageMaker MME allows time-sharing of GPU memory between SAM and LaMa in a single container. Model artifacts are compressed and stored in Amazon S3, and dynamically loaded into GPU memory when requests arrive.
* **NVIDIA Triton Inference Server with Python Backend:** Triton serves as the inference engine. Developers define the `TritonPythonModel` class structure (including `initialize` and `execute` functions) and the model configuration file (`config.pbtxt`), supporting dynamic batching and dynamic tensor shapes (e.g., `dims: [-1]`) for base64 encoded input strings.
* **API Gateway & AWS Lambda:** Serves as the API proxy layer to receive client requests, forward the payload to the SageMaker Endpoint, and return the processed image back to the client.

### Part 2: Text-Guided Inpainting & Outpainting with Amazon Bedrock

Once the object segmentation layer is established in Part 1, Part 2 extends the solution to support language-guided editing using Amazon Bedrock—a fully managed serverless service to invoke foundation models via API.

#### Language-Guided Inpainting (Replacing Objects via Text):
* **Model:** Stable Diffusion XL (SDXL 1.0) via the API ID `stability.stable-diffusion-xl`.
* **Workflow:** The payload sent to Bedrock contains the base64-encoded original image (`init_image`), the segmentation mask image (`mask_image`) generated by the SAM model, and a detailed text prompt describing the new object to be inserted (e.g., *"The Mona Lisa wearing a wig"*).
* **Control Parameters:**
  * `cfg_scale`: Controls how closely the generated image adheres to the text prompt.
  * `style_preset`: Predefines the artistic style (e.g., *digital-art, photographic, cinematic*).
  * `seed`: A random number used to initialize the image generation process to ensure reproducibility.

#### Language-Guided Outpainting (Expanding Image Canvas via Text):
* **Model:** Amazon Titan Image Generator (TIG v1) via the API ID `amazon.titan-image-generator-v1`.
* **Workflow:**
  1. **Canvas Extension:** Create a blank canvas of the target dimensions (e.g., expanding from original size to 1024x1024) and place the original image at the center.
  2. **Mask Creation:** Generate a corresponding mask where the original image area is completely black (protecting its details from being overwritten) and the extended border is white (directing the model to paint there).
  3. **API Call:** Send the payload to the Bedrock API with `taskType` set to `OUTPAINTING`.
* **Control Parameters:**
  * `text`: Text prompt describing the background context to be extended.
  * `outPaintingMode`: Configured as `DEFAULT` to blend the boundaries between the old and new image smoothly, or `PRECISE` to keep sharp borders.

### Conclusion

This hybrid architecture represents an optimal pattern for enterprises:
1. **Self-Managed Amazon SageMaker MME:** Used for open-source, medium-sized models (like SAM and LaMa) to control execution costs and handle core logic locally with low latency.
2. **Serverless Amazon Bedrock API:** Used to invoke large, complex image generation models (SDXL, Titan Image Generator) on-demand, avoiding expensive idle infrastructure costs.
